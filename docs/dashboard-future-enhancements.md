# Dashboard 后续增强记录

> 本文用于暂存首页后续可以增强的内容，不代表当前阶段立即开发。
> 当前阶段仍以「资产趋势图历史行情与月末快照优化」收尾为主，下一阶段需求由项目 owner 进一步整理后再拆分实现。

## 当前状态

首页 Dashboard 已经具备初步可用能力：

- 顶部核心指标卡片：总资产、股票市值、现金余额 / 融资负现金、净入金、总收益、收益率、已实现盈亏、已实现净收益。
- 资产趋势图：优先读取月末快照，按月末展示总资产和净入金，并支持显示股票市值、现金余额。
- 持仓占比图：展示当前持仓市值占比。
- 收益拆分、已实现盈亏、近期交易等基础分析模块。
- 月度快照计算已从 React 页面组件中移出，后端 service 负责生成和读取。
- 快照生成已预留后续增量重算方向，目前仍可接受全量生成作为当前阶段实现。

## 后续增强优先级建议

### P0：先提升数据可信度

1. **IBKR 账户价值对账**

   将 Dashboard 总资产与 IBKR 账户价值曲线进行人工或半自动对账，重点检查：

   - 月末总资产是否接近 IBKR Net Liquidation Value。
   - 融资负现金是否正确计入总资产。
   - 月末价格、现金流水、公司行动是否存在缺失。
   - 快照日期与交易日、周末、节假日的价格取值是否一致。

2. **快照异常检测**

   为月度快照增加异常提示，例如：

   - 某月总资产波动超过阈值。
   - 某只股票缺少月末价格。
   - 现金余额突然大幅变化。
   - 持仓数量为负或成本异常。

3. **Dashboard Debug 信息可视化**

   当前接口已经可以返回部分 debug 信息，后续可以在开发模式下展示：

   - 现金口径来源。
   - 价格来源。
   - 缺失价格 symbol。
   - 快照最后生成时间。

### P1：补齐收益分析口径

4. **引入更准确的收益率指标**

   当前收益率是简化口径，后续可以逐步增加：

   - MWR / XIRR：适合个人账户，因为考虑了入金和出金时点。
   - TWR：适合衡量投资组合管理表现，剔除资金流影响。
   - 当前简化收益率继续保留，但明确标注用途。

5. **收益贡献拆分**

   首页可增强为更清晰的收益来源拆解：

   - 已实现交易盈亏。
   - 未实现持仓盈亏。
   - 股息收入。
   - 手续费。
   - 汇率影响。
   - 融资利息。

6. **资产趋势图增加更完整的 tooltip**

   鼠标悬停月度点位时，可以展示：

   - 月末总资产。
   - 股票市值。
   - 现金余额 / 融资负现金。
   - 净入金。
   - 当月收益变化。
   - 缺失价格或估算数据提示。

### P1：提升首页使用体验

7. **接入 TanStack Query**

   当前首页多个接口会在刷新时重复请求，后续可以统一使用 TanStack Query 管理：

   - 请求缓存。
   - loading / error 状态。
   - staleTime 与 refetch 策略。
   - 避免 React StrictMode 下开发环境重复请求造成误解。

8. **图表筛选状态持久化**

   用户选择的趋势范围和显示指标可以保存到 localStorage，例如：

   - 1 个月 / 3 个月 / 今年 / 过去一年 / 全部。
   - 是否显示股票市值。
   - 是否显示现金余额。

9. **首页空状态和错误状态优化**

   当没有交易、没有价格、快照生成失败时，首页应提供更明确的提示：

   - 没有交易记录时引导用户导入或新增交易。
   - 缺少价格时提示缺失 symbol。
   - 快照过期时提示重新生成。

### P2：增强数据模型与导入能力

10. **真正的增量快照重算**

   当前阶段已经预留接口方向，后续可以实现：

   - 导入交易后定位最早受影响月份。
   - 从上一个月快照作为起点。
   - 只重算后续月份。
   - 对受影响账户加锁，避免并发重复生成。

11. **更完整的现金报告导入**

   目前现金口径已经尽量与 summary 保持一致，后续可以进一步结构化：

   - IBKR cash report。
   - 融资利息。
   - 股息税费。
   - 外汇兑换。
   - 账户费用。

12. **公司行动支持**

   后续需要考虑：

   - 拆股。
   - 合股。
   - 分红。
   - 股票派息。
   - symbol 变更。

13. **价格与汇率数据质量**

   后续可以增加：

   - price_history 数据补全任务。
   - 缺失价格缓存重试。
   - 非 USD 资产汇率换算。
   - 历史汇率表。

### P2：工程质量与可维护性

14. **Dashboard API 聚合优化**

   首页现在由多个接口组成，后续可以评估是否增加一个聚合接口：

   - 减少首屏请求数量。
   - 统一返回首页所需数据。
   - 保留单独接口用于模块化调试。

15. **Dashboard E2E 与截图回归**

   首页属于核心可视化页面，后续可以增加：

   - API contract test。
   - 图表数据 fixture。
   - Playwright 首页截图回归。
   - 快照生成前后数据一致性测试。

16. **后台诊断页面**

   可以在后续阶段增加一个只用于开发的诊断页面：

   - 查看快照表。
   - 查看缺失价格。
   - 手动触发快照重算。
   - 查看最近一次导入影响的月份范围。

## 暂不建议马上做的内容

以下内容不属于当前阶段，除非后续需求明确，否则先不推进：

- 真实 Gmail 集成。
- IBKR Flex Query 自动同步。
- AI 股票新闻总结。
- 登录、JWT、多用户权限。
- 股票推荐、买卖建议。
- 复杂微服务架构。
- 过早引入后台任务系统。

## 下一阶段需求整理时建议确认的问题

1. 下一阶段主目标是「数据对账准确性」还是「首页体验优化」？
2. 是否要以 IBKR Net Liquidation Value 作为总资产对账基准？
3. 是否优先实现 MWR / XIRR？
4. 是否要引入 TanStack Query 统一管理首页请求？
5. 是否要做真正的增量快照重算，还是继续保留手动全量重算？
6. 是否需要为导入流程增加“导入后自动刷新快照状态”的用户提示？
7. 是否需要先补齐测试，再继续扩展新功能？

## 面试中可以怎么描述

可以这样介绍这个阶段和后续规划：

> 我在 TradePilot 的 Dashboard 中把资产趋势计算从前端页面组件迁移到后端 service，并引入月度快照表，让首页优先读取预计算结果，避免每次打开页面都扫描全部交易记录。当前阶段先保留全量快照生成，但在接口和 service 结构上预留了从受影响月份开始增量重算的能力。后续我会围绕数据对账、收益率口径、异常检测和请求缓存继续增强，让 Dashboard 既能展示资产表现，也能帮助发现数据质量问题。
