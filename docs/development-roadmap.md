# TradePilot 开发路线图与阶段总结

更新时间：2026-05-24

本文档用于记录 TradePilot 从项目初始化到当前阶段的主要开发工作。它不是未来功能承诺清单，而是帮助回顾项目如何一步步形成当前能力边界。

当前阶段：**Email Auto Sync 邮件自动同步定时任务**

---

## 1. 项目开发原则

TradePilot 按阶段逐步开发，每个阶段只解决一个清晰的问题，避免一次性引入完整系统复杂度。

核心原则：

- 优先完成个人投资组合记录与分析闭环。
- 数据导入必须有预览、确认、去重和日志。
- 金融计算逻辑优先放在后端 service / calculator 中。
- 用户数据必须按 `userId` 隔离。
- 邮箱授权码、JWT secret、API key 等敏感信息不能返回前端或写入日志。
- 当前阶段不做股票推荐，不做 AI fallback，不做 OCR，不做复杂队列。

## 3. 已完成阶段详情

### Stage 0：项目初始化

主要目标：搭建 TradePilot 的基本工程结构。

主要工作：

- 创建 monorepo 风格的 `apps/api` 和 `apps/web`。
- 后端使用 NestJS，前端使用 React + TypeScript + Vite。
- 接入 Prisma ORM 和 PostgreSQL。
- 添加 Docker Compose 本地数据库环境。
- 建立基础健康检查接口和本地开发命令。

阶段价值：

- 形成前后端分离的基础工程。
- 为后续数据库建模、API 开发和前端页面开发提供运行框架。

---

### Stage 0.5：UI 基础框架

主要目标：建立统一的产品视觉和页面布局。

主要工作：

- 建立侧边栏导航、顶部栏、页面容器和基础卡片样式。
- 明确专业金融 SaaS Dashboard 风格。
- 使用浅色主内容区、白色卡片、细边框、绿色盈利、红色亏损。
- 建立 Dashboard、Holdings、Transactions、Imports、Settings 等页面入口。

阶段价值：

- 让后续功能可以在一致的产品框架中扩展。
- 避免每个页面重复设计布局。

---

### Stage 1：Transaction 交易记录

主要目标：支持基础交易记录管理。

主要工作：

- 建立交易相关数据结构。
- 支持买入、卖出、数量、价格、手续费、交易日期等字段。
- 增加基础 CRUD、筛选、分页和排序能力。
- 为后续持仓计算和导入流程打基础。

阶段价值：

- 形成投资组合系统的核心原始数据来源。
- 让后端开始承担真实业务数据建模和校验职责。

---

### Stage 2：Dashboard 总览

主要目标：展示投资组合核心概览。

主要工作：

- 建立 Dashboard 汇总接口。
- 展示总资产、股票市值、现金余额、净入金、总收益等核心指标。
- 增加资产趋势和持仓占比图表。
- 前端接入后端真实数据，逐步减少 mock 依赖。

阶段价值：

- 让用户进入系统后能快速理解当前组合状态。
- 建立后端聚合数据、前端展示图表的基本模式。

---

### Stage 3：Monthly Snapshot 月度快照

主要目标：沉淀组合历史状态，支持趋势分析。

主要工作：

- 增加 portfolio monthly snapshot 和 position monthly snapshot。
- 支持按月份重建资产快照。
- 保存月末资产、现金、市值、净入金等数据。
- 为资产趋势、收益分析和后续指标计算提供基础。

阶段价值：

- 将“当前计算”扩展为“历史可追踪”。
- 避免每次图表都从所有交易记录临时重算。

---

### Stage 4：Holdings 持仓

主要目标：准确计算当前持仓。

主要工作：

- 按交易记录计算当前股票数量、平均成本、成本金额。
- 接入行情数据后计算市值、未实现盈亏。
- 处理买入、卖出、转入、转出、拆股等对持仓有影响的事件。
- 修复和验证持仓数量、成本、现金余额等计算问题。

阶段价值：

- 建立组合分析最核心的金融计算能力。
- 让交易导入结果可以落到真实持仓变化上。

---

### Stage 5：Transactions 明细

主要目标：增强交易数据审计和排查能力。

主要工作：

- 新增交易明细页面。
- 展示交易日期、symbol、side、quantity、price、fee、amount 等字段。
- 保留 `rawRecord` 等源数据可见性，方便排查导入和解析问题。
- 保留 IBKR 源字段中的 `realizedPnl`，避免不必要地用 fallback 覆盖。

阶段价值：

- 让用户可以追踪每一条交易从导入到展示的链路。
- 为后续排查导入错误和持仓错误提供依据。

---

### Stage 6：Cash Flow 资金流水

主要目标：只管理入金和出金。

主要工作：

- 建立独立 `cash_flows` 数据表。
- 从导入确认流程中同步写入 Deposit / Withdrawal。
- 前端页面只展示入金、出金、净入金和现金流水。
- 支持资金流水导出。

阶段价值：

- 将资金流水从交易明细中拆出来，降低页面复杂度。
- 让净入金、收益率和现金余额计算更清晰。

---

### Stage 7：Performance 收益分析

主要目标：展示组合收益表现。

主要工作：

- 增加 Performance 页面。
- 展示收益、收益率、资产趋势、月度表现等分析视图。
- 复用月度快照和 Dashboard 聚合数据。
- 将核心统计逻辑放在后端 service / calculator 中。

阶段价值：

- 从“记录资产”推进到“分析表现”。
- 为投资复盘提供更接近真实产品的分析页面。

---

### Stage 8：Trading Behavior 交易行为

主要目标：分析用户交易习惯和行为模式。

主要工作：

- 新增 Trading Behavior 页面。
- 统计交易次数、买卖分布、持仓周期、交易频率等指标。
- 支持按时间范围查看行为数据。
- 用图表展示交易行为和盈亏分布。

阶段价值：

- 帮助用户从交易记录中看到自己的行为特征。
- 展示后端聚合分析和前端可视化能力。

---

### Stage 9：Import 管理

主要目标：建立可靠的 IBKR CSV 导入工作流。

主要工作：

- 建立 CSV 导入预览接口。
- 解析 IBKR CSV，生成标准化交易记录预览。
- 使用 `sourceHash` 做重复检测。
- 用户确认后才写入交易、资金流水和导入日志。
- 增加 ImportJob / ImportRecord 历史列表、详情和日志删除能力。

阶段价值：

- 完成“外部数据 → 预览 → 确认 → 入库 → 审计”的导入闭环。
- 为后续邮件 PDF 导入和自动同步复用导入日志模型。

---

### Stage 10：User 登录与数据隔离

主要目标：引入用户体系和数据隔离。

主要工作：

- 新增 User 模型。
- 支持注册、登录、JWT 鉴权。
- 业务接口从 token 中解析当前用户。
- 为 import jobs、transaction events、cash flows、snapshots 等数据补充 `userId`。
- 将本地历史数据迁移到默认 demo 用户。

阶段价值：

- 从单用户脚本式系统升级为具备多用户隔离基础的应用。
- 为邮箱设置、用户级导入配置和个人数据保护打基础。

---

### Stage 11：Settings 设置

主要目标：集中管理用户偏好和系统配置。

主要工作：

- 新增 Settings 页面。
- 支持个人资料设置。
- 支持导入设置，例如导入后是否刷新行情、重建快照、重算指标。
- 支持行情数据设置和 symbol mapping。
- 新增系统状态接口，用于展示数据数量、最近导入时间、最近行情更新时间等。

阶段价值：

- 将系统配置从代码常量逐步迁移到用户级设置。
- 为后续邮箱设置和自动同步配置提供入口。

---

### Stage 12：Email 设置

主要目标：支持 QQ / 163 邮箱 IMAP 连接配置。

主要工作：

- 在 Settings 中新增 Email 设置。
- 第一版采用 provider-driven UX，只让用户填写邮箱服务商、邮箱地址和授权码。
- 服务端根据 QQ / 163 自动选择 IMAP host、port 和 secure。
- 授权码使用 `EMAIL_SECRET_KEY` 加密保存。
- 前端只显示 `hasAuthSecret`，不返回授权码。
- 支持连接测试和断开连接。

阶段价值：

- 完成真实邮箱接入前的安全基础。
- 明确邮箱配置必须用户隔离、凭证加密、不能打印授权码。

---

### Stage 13：Email Scan 邮件扫描

主要目标：只读扫描 IBKR 邮件和 PDF 附件。

主要工作：

- 在 Imports 页面增加 Email Import Tab。
- 支持扫描最近 3 / 7 / 30 / 90 天邮件。
- 匹配 IBKR Daily Trade Report 邮件。
- 读取 PDF 附件元数据，记录 messageId、attachmentHash、subject、from、date 等。
- 不标记已读、不删除、不移动邮件。

阶段价值：

- 先验证邮箱扫描链路，不急于自动入库。
- 为 PDF 解析和邮件去重建立基础。

---

### Stage 14：Email PDF Import

主要目标：解析 IBKR Daily Trade Report PDF 并确认导入。

主要工作：

- 提取 PDF 文本。
- 使用规则解析 Daily Trade Report 中的交易表格。
- 生成 Email PDF trade preview。
- 标记 NEW / DUPLICATE / ERROR。
- 用户确认后写入交易记录。
- 复用 `sourceHash` 去重、ImportJob / ImportRecord 导入日志。
- 修复 PDF 交易币种、费用、sourceHash 等解析细节。

阶段价值：

- 从“邮件扫描”推进到“邮件 PDF 交易导入”。
- 不使用 AI、不使用 OCR，保持规则解析可解释、可调试。

---

### Stage 15：Email Auto Sync 邮件自动同步

主要目标：每天早上 7 点自动同步已连接邮箱。

主要工作：

- 引入 NestJS Schedule。
- 每天 07:00 按项目时区执行邮件同步任务。
- 查询已开启自动同步且状态为 CONNECTED 的邮箱。
- 默认扫描最近 3 天 IBKR 交易报告邮件。
- 自动读取 PDF 附件并复用规则解析。
- NEW 交易自动入库。
- DUPLICATE 跳过。
- ERROR 写入任务日志，不自动入库。
- 同步完成后根据用户导入设置刷新行情、重建快照、重算指标。
- 新增 EmailSyncJob 的 triggerType、insertedCount、duplicateCount、errorCount 等统计字段。
- Imports / Email Import 页面展示手动同步和定时同步历史。
- Settings 页面展示自动同步开关、同步时间、扫描范围、最近同步状态。
- 右上角 Sync 时间改为读取最新数据同步时间：CSV 导入时间和邮件同步时间两者取最新。

阶段价值：

- 完成从手动导入到自动同步的关键升级。
- 建立第一版轻量定时任务，不引入 BullMQ / Redis。
- 保持邮件只读、安全、不破坏用户邮箱状态。

## 4. 当前系统主要能力

### 4.1 数据记录能力

- 记录股票交易事件。
- 记录入金和出金。
- 保存导入文件、导入任务和导入记录。
- 保存邮件扫描记录和邮件同步任务。

### 4.2 投资组合计算能力

- 计算当前持仓。
- 计算现金余额、净入金、股票市值、总资产。
- 计算已实现盈亏、未实现盈亏和收益表现。
- 生成月度资产快照和持仓快照。

### 4.3 数据导入能力

- IBKR CSV 导入。
- IBKR 邮件 PDF 扫描。
- IBKR Daily Trade Report PDF 规则解析。
- 导入预览、确认、重复检测和错误日志。
- 邮件自动同步。

### 4.4 设置和安全能力

- 用户注册登录和 JWT 鉴权。
- 用户级数据隔离。
- 用户级导入设置、行情设置、邮箱设置。
- 邮箱授权码加密保存。
- 邮件扫描不标记已读、不删除、不移动。

## 6. 后续候选方向

后续阶段可以按优先级选择，不建议一次性全部展开。

### 6.1 自动同步增强

- 将定时任务升级为 BullMQ + Redis。
- 增加失败重试、任务超时、任务取消。
- 增加多实例部署下的分布式锁。
- 支持用户自定义同步时间和扫描范围。

### 6.2 数据质量和审计增强

- 增加导入差异对比。
- 增加解析失败样本的调试面板。
- 增加 sourceHash / attachmentHash / messageId 的排查入口。
- 增加导入回滚策略。

### 6.3 行情和指标增强

- 优化行情缓存和缺失 symbol 处理。
- 增加更完整的收益率计算口径说明。
- 增加 TWR / MWR 等更标准的投资表现指标。

### 6.4 部署和运维

- 整理生产环境变量说明。
- 增加数据库备份策略。
- 增加部署检查清单。
- 增加基础监控和错误日志归档。

## 7. 面试中如何描述这个项目

可以这样概括：

> TradePilot 是一个个人股票投资组合记录与分析平台。我按阶段从基础交易记录、持仓计算、资金流水、Dashboard 图表逐步做到 IBKR CSV 导入、用户数据隔离、邮箱 IMAP 接入、PDF 交易报告解析和每天定时自动同步。后端使用 NestJS + Prisma + PostgreSQL，核心难点是金融数据建模、导入幂等性、sourceHash 去重、用户级数据隔离、邮件授权码加密，以及自动同步任务的日志和错误处理。前端使用 React + TypeScript，把导入预览、同步历史、Dashboard 指标和设置页做成完整的产品工作流。

重点能力：

- 能从真实业务问题拆阶段开发。
- 能设计可审计的数据导入流程。
- 能处理金融数据计算和精度风险。
- 能用 NestJS / Prisma 建模复杂业务关系。
- 能在前端构建清晰的数据管理和分析界面。
- 能考虑凭证安全、用户隔离和任务幂等性。
